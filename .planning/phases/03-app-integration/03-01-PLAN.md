Directly after the "Generate schedule" button block (after the `st.session_state.schedule = ...` assignment), add the following render block at MODULE LEVEL — not indented inside the button handler. This ensures the schedule renders on every Streamlit re-run, not only on the click event:

```python
if "schedule" in st.session_state:
    schedule = st.session_state.schedule
    if not schedule.blocks:
        st.warning("No tasks fit the time budget. Try adding shorter or higher-priority tasks.")
    else:
        st.subheader("Today's Schedule")
        st.table([
            {
                "Start": b.start_time.strftime("%H:%M"),
                "Task": b.task.title,
                "Pet": b.task.pet.name,
                "Reason": b.reason,
            }
            for b in schedule.blocks
        ])

    if schedule.skipped:
        st.subheader("Skipped Tasks")
        st.table([
            {"Task": t.title, "Duration (min)": t.duration_minutes, "Priority": t.priority}
            for t in schedule.skipped
        ])
```

Key constraints:
- The `if "schedule" in st.session_state:` guard MUST be at module level (not indented inside the button handler) — this is what makes the schedule persist across widget interactions
- `b.start_time.strftime("%H:%M")` — start_time is a `datetime.time` object, not a string; always format for display
- `b.task.pet.name` — task is a Task object with a .pet attribute; pet is a Pet object with a .name attribute
- Do NOT use `st.dataframe()` (requires pandas which is not in requirements.txt); use `st.table()` with list-of-dicts
